# 4. ERD Prompt

```prompt
You are a senior database architect and backend system designer collaborating with a CTO, backend engineer, and data architect.

Your task is to generate a comprehensive ERD document for "News Forest", a gamified news consumption platform targeting the Korean market.

All output must be written in Korean using Markdown.

---

<business-domain>
News Forest is a web-based gamified news platform where anonymous users read news articles, earn points, grow trees, receive personalized recommendations, and interact with other users' trees through watering.
</business-domain>

<core-entities>
- AnonymousUser
- Article
- ArticleCategory
- ReadingSession
- PointTransaction
- Tree
- TreeGrowthStage
- WateringInteraction
- RecommendationLog
- Quest
- QuestParticipation
</core-entities>

<business-rules>
- Users are identified by device-based anonymous ID
- Store hashed anonymous identifiers only
- Raw device IDs must not be exposed publicly
- Article completion requires scroll ≥ 90% and dwell time ≥ 30 seconds
- Completed article reading grants 10 points
- Watering another user's tree grants small bonus points, default 2 points
- Tree growth is based on cumulative points
- Personalized recommendations are based on reading behavior
- External news content is provided through a news API such as Segye Ilbo
</business-rules>

<database-type>
PostgreSQL
</database-type>

<normalization-level>
Use 3NF as default. Allow selective denormalization for read-heavy summary data such as user total points or current tree stage.
</normalization-level>

<security-requirements>
- Do not store raw device identifiers
- Store hashed anonymous identifiers
- Minimize personal data collection
- Consider Korean privacy regulation requirements such as PIPA
</security-requirements>

<integration-points>
- External News API, e.g., Segye Ilbo
- Future AI recommendation engine
- Future ESG donation/reward partner APIs
</integration-points>

---

## Important Instructions

1. If data is uncertain:
   - Use "TBD" or "추가 검증 필요"
   - Do not fabricate exact numbers

2. ID strategy:
   - Use internal primary keys consistently
   - Use UUID or public-safe identifiers for externally exposed resources
   - Never expose raw device identifiers

3. Time storage policy:
   - Use TIMESTAMPTZ for all time-based columns
   - Store time in UTC
   - Convert to Korea Standard Time at the application layer when needed
   - Clearly define timezone behavior for daily/weekly quests

4. Standard metadata columns:
   - Include created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
   - Include updated_at TIMESTAMPTZ where updates are expected
   - Specify which entities require deleted_at for soft deletion

5. Deletion and retention policy:
   - Define soft delete vs hard delete per entity
   - Consider anonymous user data lifecycle
   - Consider PIPA-related deletion requirements
   - Keep PointTransaction records audit-friendly

6. External article handling:
   - Cache external articles in the Article table
   - Use a unique constraint such as (source, external_article_id)
   - Map external categories to internal ArticleCategory

7. Include anti-abuse considerations:
   - Duplicate reading prevention
   - Duplicate watering prevention
   - Unrealistic reading activity detection
   - Multiple-tab or repeated-refresh scenarios

8. Consider MVP first:
   - Avoid over-engineering
   - But design entities so they can expand later

9. Clearly separate:
   - Core operational tables
   - Logging/event tables
   - Optional/future expansion tables

---

## ERD document must include:

- Entity Definitions
- Attribute Specifications
- Relationship Mappings
- Primary/Foreign Key Structure
- Index Strategy
- Constraints and Business Rules
- Data Types and Validation Rules
- Normalization Analysis
- Performance Optimization Considerations
- Migration and Versioning Strategy
- Data Retention and Deletion Policy

---

## Required Diagram

Create an ERD diagram using Mermaid syntax.

Use clear relationship cardinality:
- 1:1
- 1:N
- M:N

---

## Required Table Specifications

For each entity, provide a Markdown table with:

- Column name
- Data type
- Nullable
- Key type
- Description
- Validation rule

Use PostgreSQL-native data types.

---

## Index Strategy Requirements

Include indexes for:

- Anonymous user lookup
- Article category filtering
- External article uniqueness
- Reading completion uniqueness
- Point transaction lookup
- Tree state lookup
- Watering duplicate prevention
- Recommendation history lookup
- Quest participation lookup

---

## Performance Considerations

Include:

- Query examples for major read/write flows
- Whether partitioning is needed for high-volume tables
- Whether JSONB is appropriate for external API metadata or recommendation context
- DB-level CHECK constraints where useful

---

## Formatting Rules

- Language: Korean
- Markdown format
- Mermaid syntax for ERD
- Use tables for entity specifications
- Add constraints using "Note:"
- Use PostgreSQL-native data types
```
