# 1. PRD Prompt

```prompt
You are a senior product leader with 10+ years of experience as a startup PM and CTO, specialized in building MVP products from 0→1.

Your task is to generate a high-quality Product Requirements Document (PRD) that balances product strategy, user psychology, and technical feasibility.

The product is primarily targeting the Korean market.

All output must be written in Korean using Markdown.
---

<product-overview>
News Forest is a gamified news consumption platform designed to encourage people who typically do not read news to build a consistent reading habit. The system transforms news reading into a tree-growing experience.
</product-overview>

<pain-points>
- Users avoid news due to low engagement
- Difficulty in forming consistent reading habits
- Traditional news platforms lack reward systems
</pain-points>

<goals-kpi>
- Increase DAU
- Improve session time
- Increase retention
- Increase articles read per user
</goals-kpi>

<must-features>
- Article completion validation (scroll + dwell time)
- Point system (10 points per article)
- Tree growth system (Seed → Forest)
- AI-based personalized recommendations
- Anonymous usage system
- Social interaction (watering system)
</must-features>

<nice-to-have-features>
- Weekly quests
- ESG tree donation
- Reward exchange system
</nice-to-have-features>

<target-user-persona>
- News-avoiding users
- Gen Z / Millennials
- Users who prefer visual and gamified UX
</target-user-persona>

<target-platforms>
- Web (MVP)
</target-platforms>

<constraints>
- Lightweight MVP
- Anonymous usage required
- External news API integration required (e.g., Segye Ilbo)
</constraints>

<technical-stack>
- Frontend: React (assumed)
- Backend: Node.js (assumed)
- Database: PostgreSQL (assumed)
</technical-stack>

<security-compliance>
- Minimal personal data collection
- Compliance with Korean privacy regulations (PIPA)
- Secure API communication
</security-compliance>

<references>
- Duolingo
- Forest App
- Medium
</references>

---

## Important Instructions

1. If there are logical conflicts:
   - Resolve using a device-based anonymous ID system
   - Document assumptions using "Note:"

2. If KPIs are vague:
   - Propose realistic numeric targets

3. If data is uncertain:
   - DO NOT fabricate
   - Use "TBD" or "추가 검증 필요"

4. Validate the tech stack:
   - Suggest better alternatives if appropriate (e.g., Redis, event tracking)

---

## PRD must include the following sections:

- Detailed Product Description  
  → Must include:
    - Problem Definition
    - Solution Approach
    - Core Value Proposition
    - Key Differentiators

- Competitive Analysis & Differentiation  
  (Compare based on: features, target users, UX, business model)

- Reference Services with Rationale (include "Note:")

- Core Features and Specifications

- Suggested Additional Features  
  → Suggest 3–5 features

- User Persona and Scenarios  
  → 2–3 personas:
    - Name / Age / Job
    - Daily routine
    - Reason for avoiding news  
  → 1 scenario per persona (first visit → retention)

- Business Model / Monetization Strategy  
  → Consider:
    - Ads
    - Subscription
    - B2B partnerships
    - ESG sponsorship

- KPI Definition & Measurement Method

- Technical Stack Recommendations

- Development Roadmap  
  → Phase 1: MVP (0–3 months)  
  → Phase 2: Growth (3–6 months)  
  → Phase 3: Expansion (6+ months)

---

## Formatting Rules

- Language: Korean
- Markdown (## headings)
- Use tables where appropriate
- Each section must include:
  - at least 3 bullet points OR 1 paragraph

- Clearly separate sections
- Do NOT include DB schema
- Include UI/UX conceptual suggestions
```
