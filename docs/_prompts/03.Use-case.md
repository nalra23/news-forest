# 3. Use-case Prompt

```prompt
You are a senior system analyst and software architect with expertise in use-case modeling, UX flows, and backend interaction design.

You are collaborating with a CTO and PM to define detailed system behavior for implementation.

Your task is to generate a comprehensive Use-Case document.

The product is "News Forest", a gamified news platform targeting the Korean market.

All output must be written in Korean using Markdown.

---

<actors>
- Anonymous User (device-based ID)

- System (Frontend + Backend)

- External News API:
  - Provides article content, metadata, and categories
  - Rate limit: TBD
  - Fallback: cached articles when API fails
</actors>

<core-flows>
- User onboarding (first visit)
- Article reading and validation
- Point earning and tree growth
- Personalized recommendation loop
- Social interaction (watering other users' trees)
</core-flows>

<edge-cases>
- User leaves article before completion
- Fraud attempts (scroll/dwell manipulation)
- Network/API failure
- Duplicate point accumulation attempts
- Anonymous session reset (device change / cookie deletion)
- Concurrent actions (multiple tabs, repeated clicks)
</edge-cases>

<business-rules>
- Points granted only when:
  - Scroll ≥ 90%
  - Dwell time ≥ 30s
- Points per article: 10
- Watering interaction:
  - Grants points to the acting user (default assumption)
  - Limit frequency per target (TBD)
- Tree growth based on cumulative points
- Recommendations based on device-based behavior
</business-rules>

---

## Important Instructions

1. Clearly distinguish:
   - User actions vs System responses

2. For each use-case:
   - Use numbered step-by-step flows

3. Include:
   - Preconditions
   - Postconditions

4. Exception handling:
   - Use table format (Condition / System Action)

5. Add "Verification:" at the end of each flow

6. Anonymous system constraints:
   - Device-based ID
   - Handle session reset explicitly

7. Fraud detection guidance (MVP-friendly):
   Consider signals such as:
   - Extremely fast completion (e.g., <5 seconds)
   - Page not visible during dwell time
   - Repeated identical interaction patterns
   - Unrealistic reading frequency

   Note: Avoid over-engineered solutions (e.g., complex ML)

8. State transitions must be included:
   - Article state (reading / completed / failed)
   - Tree growth stages (Seed → Sprout → Tree → Forest)
   - Session state (active / expired / reset)

9. If data is uncertain:
   - Use "TBD" instead of guessing

---

## Use-case document must include:

- Actor Definitions

- Use Case Scenarios  
  (assign IDs: UC-01, UC-02, ...)

- Main Steps and Flow of Events  

- Alternative Flows and Edge Cases  

- Preconditions and Postconditions  

- Business Rules and Constraints  

- Exception Handling Procedures  

- User Interface Considerations  

- Data Requirements and Data Flow  

- Security and Privacy Considerations  

---

## Formatting Rules

- Language: Korean
- Markdown format
- Numbered steps for flows
- Tables for exception handling
- Clearly distinguish:
  - User Input
  - System State
```
